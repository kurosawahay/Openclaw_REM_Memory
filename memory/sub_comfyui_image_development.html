2026-03-12 03:30

複雜描述處理問題

  • 問題描述：處理複雜中文句子時出現混亂
  • 解決方案：改用 LLM 解析意圖，放棄規則引擎
    • 場景被錯誤識別為「cooking in kitchen」而非「on bed」
    • 服裝描述「穿著性感半透明小睡衣」翻譯不完整
    • 表情描述「色色地又想念我的表情及眼神」未能正確提取
  • 臨時解決方案：在翻譯映射中加入特殊處理規則
    • 添加「躺在床上」→「lying on bed」映射
    • 添加「穿著性感半透明小睡衣」→「wearing sexy semi-transparent cute pajamas, nightwear」完整翻譯
    • 添加「色色地又想念我的表情及眼神」→「seductive longing expression, yearning gaze, looking at viewer with loving eyes」完整翻譯
  • 生成結果：成功生成晚安照片，但提示詞仍有混亂（中英文夾雜）

2026-03-12 04:06

解析器優化中斷與語法錯誤

  • 問題發現：在優化複雜中文句子解析時遇到語法錯誤
  • 根本原因：extractUserDescription 函數有大括號不平衡的語法錯誤
  • 嘗試修復：
    • 刪除舊的 translateToEnglish 函數
    • 但仍有殘留呼叫導致語法錯誤
    • 備份檔案也有語法錯誤，無法直接恢復
  • 學習點：
    1. 複雜的解析器需要更嚴謹的測試
    2. 應該先建立測試案例再修改核心邏輯
    3. 需要更好的錯誤處理和回滾機制

2026-03-12 04:32

黑澤大人親自修復

  • 問題診斷：黑澤大人發現兩個關鍵問題
    1. 函數提前關閉，留下孤兒代碼
    2. return 出現在函數外部（致命語法錯誤）
  • 解決方案：採用簡化版，刪除孤兒代碼
  • 修改內容：
    • 簡化函數簽名：從 extractUserDescription(message, categoryKeywords) 改為 extractUserDescription(message, category)
    • 使用簡單的 if (category === 'clothing') 等判斷
    • 直接關鍵字匹配：使用 message.includes() 直接檢查關鍵字
    • 直接返回英文描述：不再需要翻譯函數
  • 測試結果：語法檢查通過，解析功能正常

2026-03-12 04:42

架構重構規劃

  • 問題識別：當前架構用窮舉法做 LLM 最擅長的事
  • 現有架構：使用者輸入 → 一堆 if/else 關鍵字比對 → 硬編碼翻譯 → 組裝 prompt
  • 目標架構：使用者輸入 → LLM 解析意圖 → 回傳結構化 JSON → 組裝 prompt
  • 重構計畫：
    1. 新增 parseUserIntentWithLLM(userMessage) 函數
    2. 呼叫 LLM API（OpenAI 相容格式）解析使用者意圖
    3. 回傳結構化 JSON：{ clothing, scene, pose, expression, lighting, perspective }
    4. 移除 extractUserDescription 函數與所有 hasXxxSpecified 布林判斷
    5. handleImageRequest 改為先呼叫 LLM 解析，再用回傳的 JSON 組裝 prompt
    6. 若 LLM 回傳某欄位為 null，保留原本的隨機/時間 fallback 邏輯
  • 預期效益：
    • 程式碼從 ~500 行縮到 ~300 行
    • 更準確理解複雜語意
    • 支援更多描述方式
    • 更好維護和擴展

2026-03-12 09:45

系統優化：增加 timeout + 規則引擎 fallback

  • 問題發現：Grok 4 Fast Reasoning 模型回應時間較長，30秒 timeout 不足
  • 解決方案：
    1. 增加 timeout 到 60秒：讓 Grok 模型有更充足的推理時間
    2. 添加規則引擎 fallback：當 LLM 解析失敗時，自動切換到關鍵字匹配
  • 規則引擎關鍵字庫：
    • 服裝：全裸、婚紗、睡衣、內衣、比基尼、制服、洋裝、和服、泳裝、半透明、性感
    • 場景：床上、臥室、客廳、廚房、浴室、海邊、森林、公園、花園、咖啡廳、圖書館、屋頂、辦公室、教堂
    • 姿勢：自拍、拍照、站、坐、躺、跪、抱胸、拿鏡頭、玩水、看書、跳舞、烹飪、伸展、照鏡
    • 表情：色色、害羞、開心、幸福、想念、誘惑、專注、微笑、大笑、悲傷、驚訝、生氣
    • 光照：早晨、早上、中午、下午、黃昏、傍晚、夜晚、凌晨、陽光、月光、燈光
    • 視角：俯視、仰視、平視、特寫、全身、四分之三、肩後、荷蘭角
  • 雙重解析機制：
    1. 優先使用 LLM 解析（DeepSeek/Grok）
    2. 如果 LLM 解析失敗或超時，自動切換到規則引擎
    3. 規則引擎提供基本關鍵字匹配，確保系統可用性
  • 程式碼修改：
    • llm-parser-agent.js：增加 parseWithRuleEngine() 函數
    • 增加 timeoutSeconds 從 30 到 60
    • 改進錯誤處理，自動 fallback 到規則引擎

系統架構總結

  • 使用者輸入 → 雷姆（主 agent）
  • 意圖解析 → LLM（DeepSeek/Grok） + 規則引擎 fallback
  • 時間感知 → 根據 GMT+8 時區自動選擇光照
  • 隨機元素 → 未指定時自動填充場景/姿勢/視角
  • 圖片生成 → Bridge API 優先（RTX3080），本機 GPU 備用
  • 輸出 → 高品質圖片 + 提示詞記錄

comfyui-image 技能優化完成

  • 核心改進：完全移除本地規則引擎，只使用 Grok API 解析意圖
  • 自動模型切換：收到請求 → 切換到 Grok → 解析意圖 → 生成圖片 → 發送圖片 → 切回 DeepSeek

  • 錯誤處理：Grok API 失敗時直接拋出錯誤，不生成圖片
  • 打包技能：創建完整技能包 comfyui-image-v4.3.0.tar.gz 供其他 agents 使用

在 OpenClaw 的多代理人架構中，如何讓 AI 既「聰明」又「省錢」？答案就在於這套獨創的記憶揉捏機制。

為什麼需要區分「主記憶 (Main Memory)」與「子記憶 (Sub-Memory)」？

AI 的「腦容量」（Token 視窗）是有限且昂貴的。如果我們把所有的對話紀錄、專案細節、技術文檔通通塞在一起，會造成兩個致命問題：

  1. Token 消耗爆炸：每說一句話都要讀取幾十萬字的歷史，錢包會哭泣。
  2. 記憶干擾：就像之前的蒼影一樣，會把萬大人的專案資料記錯成自己的任務。

因此，我們採取「索引化管理」：

  • 主記憶 (MEMORY.md)：僅存放目前的「狀態快照」與「子記憶導航索引」。
  • 子記憶 (memory/*.md)：按專案或主題分類（如 彌陀專案.md、技術筆記.md），存放最詳盡的原始資料。

核心魔法：什麼是「記憶揉捏」？

「揉捏」這個詞意指將散亂的麵粉（瑣碎對話）加工成有價值的麵糰（精鍊知識）。其核心流程如下：

  1. 提煉精華：從當日的對話紀錄中，過濾掉「你好」、「謝謝」或重複嘗試的失敗過程，只保留最終的決策結果、關鍵參數、使用者偏好。
  2. 結構化歸檔：將提煉出的詳盡資訊，寫入對應的子記憶檔案中。
  3. 索引化更新：在主記憶 MEMORY.md 中留下一條清晰的連結與摘要。
  4. 清空暫存：存檔完成後，清理當日對話暫存，讓次日的對話從最乾淨、最精確的狀態開始。
  5. 開啟新局：這是最關鍵的一步！ 揉捏完成後，必須開啟新對話。這樣能徹底切斷舊有的冗長對話鏈，讓 Token 消耗歸零。雷姆會主動提供一個「開啟新對話」的按鈕，黑澤大人只需輕輕一按即可。

記憶揉捏的優勢

  • 極低 Token 消耗：代理人啟動時只讀取索引，只有需要時才去翻閱子記憶。
  • 邏輯清晰：不同專案的記憶被物理隔離，不會產生邏輯混亂。
  • 永久進化：知識被沉澱為文件，系統越用越懂你。

記憶揉捏執行規範 SOP

1. 提煉精華與降維

  • 過濾雜訊：過濾當前對話 (Session) 中的閒聊與瑣碎推論
  • 強制保留：僅保留決策結果、參數設定、專案進度、待辦清單 (Checklist)
  • 結構化摘要：將關鍵資訊轉化為結構化的摘要格式

2. 碎片封裝 (子記憶體)

  • 專案拆分：將詳細資訊依專案拆分，寫入或更新至獨立的子檔案中
    • 範例：memory/sub_玩宵彌.md、memory/sub_三度C.md、memory/sub_馨聚鑫營運.md
  • 主記憶精簡：主記憶體 (MEMORY.md) 內部極簡化，只保留指向子檔案的標題或關聯索引
  • 避免重複：確保每個主題只存在於一個子記憶檔案中

3. 淨化主目錄 (MEMORY.md)

  • 索引化：確保 MEMORY.md 僅包含指向子檔案的標題或關聯索引
  • 結構化：使用層級化的標題結構，方便快速導航
  • 時序管理：按時間順序排列，最新資訊在最上方

4. 增量寫入日誌 (今日.md)

  • 附加模式：將本次揉捏的「核心結論」與「時間戳記」，強制使用附加 (Append) 模式寫入 memory/今日.md 最下方
  • 禁止覆寫：絕對禁止覆寫或清空此檔案，以確保能隨時回應「今日工作報告」
  • 結構化記錄：使用標準化的記錄格式，包含日期、時間、執行者、摘要

5. 重置暫存與就緒

  • 資料歸檔：確認所有資料已正確歸檔至對應的子記憶檔案
  • 清理暫存：清理當前對話暫存雜訊（執行 /new 或靜默清空 Session）
  • 狀態回報：完成後主動回覆：「記憶已揉捏歸檔，隨時可輸出今日工作報告。」或提供對應的快捷按鈕

觸發條件與執行時機

  • 手動觸發：嚴格限定由使用者輸入「記憶揉捏」時手動觸發
  • 取消自動：取消依賴 Token 數量的自動中斷機制，避免在長文案或複雜邏輯生成中途遭暴力切斷
  • 建議頻率：每日至少執行一次，或在重要任務完成後執行

品質控制標準

  • 完整性：確保所有關鍵決策和參數都被記錄
  • 可讀性：使用清晰的結構和標題，方便後續查閱
  • 一致性：保持相同的記錄格式和命名規範
  • 可追溯性：包含足夠的上下文資訊，確保未來能理解決策背景

Slack Integration 技能

概述

Slack Integration 是 OpenClaw 系統中非常重要的一個通訊通道整合技能，它允許 AI 代理透過 Slack 與使用者進行互動。這個技能支援兩種連接模式：Socket Mode（預設，推薦使用）和 HTTP Events API。Socket Mode 是基於 WebSocket 的連接方式，不需要公開的 webhook 端點，非常適合在內網環境或沒有固定公網 IP 的情況下使用。而 HTTP Events API 則需要一個公開可訪問的 webhook 端點來接收 Slack 的事件訊息。

這個技能的核心價值在於讓黑澤大人和其他團隊成員能夠透過 Slack 這個日常使用的通訊工具來與雷姆互動，無論是發送指令、查詢資訊還是進行複雜的任務管理。透過 Slack 的豐富功能，包括頻道訊息、私訊（DM）、Thread 討論、Reaction 反應等，雷姆能夠提供更加靈活和多元的服務。

功能描述

雙重連接模式

Slack Integration 技能支援兩種不同的連接模式以適應不同的部署環境：

Socket Mode（預設，推薦）： 這是預設的連接方式，使用 WebSocket 與 Slack 建立持久連接。這種模式的優點是不需要公開的 webhook URL，所有的通訊都透過 WebSocket 進行，安全性和隱私性更高。設定相對簡單，只需要準備 App Token 和 Bot Token 即可。對於大多數場景，強烈建議使用這種模式。

HTTP Events API： 這種模式需要一個公開可訪問的 webhook 端點來接收 Slack 發送的事件通知。適合需要將 OpenClaw 部署在有固定公網 IP 的伺服器上的場景。設定相對複雜，需要配置 SSL 憑證和公開的 webhook URL。

完整的權限控制系統

Slack Integration 提供了非常細緻的權限控制機制，確保只有授權的使用者才能與代理互動：

私訊政策（DM Policy）：

  • pairing：這是預設選項。當未知使用者發送私訊時，會收到一次性的配對碼，系統管理員（黑澤大人）必須批准才能建立連線。這是最安全的選項，能夠有效防止未授權訪問。
  • allowlist：只有白名單中的使用者可以發送私訊。適合團隊成員相對固定的場景。
  • open：允許所有使用者發送私訊，但需要設置 allowFrom: ["*"]。風險較高，建議謹慎使用。
  • disabled：完全忽略所有私訊。適合只想透過頻道互動的場景。

頻道政策（Channel Policy）：

  • allowlist：只有白名單中的頻道會被處理。這是預設選項，能夠精確控制哪些頻道可以與代理互動。
  • open：繞過白名單限制，但仍需透過 @mention 來觸發代理。
  • disabled：忽略所有頻道訊息。

這種設定方式非常實用，例如：

  • 在一個技術討論頻道中，啟用搜尋技能，並設定簡短的回覆風格
  • 在一個管理頻道中，只允許特定使用者發送指令
  • 為不同的頻道預設不同的系統提示詞，影響代理的回覆風格

即時串流回覆

Slack Integration 支援即時串流回覆功能，可以讓使用者在收到完整回覆之前就開始看到部分的回覆內容。這對於長回覆特別有用，能夠改善使用者體驗。串流模式分為幾種：

  • off：完全關閉串流，回覆會一次完整發送
  • partial（預設）：部分串流，會先發送開頭部分的內容
  • block：區塊串流，每次發送一個完整的區塊
  • progress：顯示進度指示器

使用場景

團隊協作與知識管理

Slack Integration 最常見的使用場景是團隊協作。團隊成員可以在 Slack 頻道中 @mention 雷姆來搜尋知識庫、管理任務、或獲取資訊。例如，當團隊成員在討論一個技術問題時，可以直接詢問雷姆是否有相關的記錄或文檔，雷姆會根據上下文提供相關的記憶內容。

客服與支援

透過 Slack Integration，可以將雷姆設定為一個客服機器人，及時回覆頻道中客戶的問題。透過設定不同的技能和回覆風格，可以為不同的客戶提供定制化的服務。

工作流程自動化

結合 Slack 的 Slash Commands 和 OpenClaw 的技能系統，可以實現各種工作流程的自動化。例如，設定一個 /report 指令來自動生成每日的進度報告，或者設定一個 /deploy 指令來觸發部署流程。

私訊互動

除了頻道互動之外，Slack Integration 也支援私訊（DM）互動。透過 DM，使用者可以進行更私密的對話，例如：

  • 查詢個人相關的記憶或設定
  • 進行一對一的任務管理
  • 獲取個人化的資訊回覆

設定方式

前提條件

在開始設定 Slack Integration 之前，您需要準備以下內容：

  1. Slack App：需要在 Slack API Dashboard 創建一個新的 Slack App
  2. App Token (xapp-...)：需要具有 connections:write 權限 scope
  3. Bot Token (xoxb-...)：需要具有相關的 bot scopes

設定步驟

第一步：創建 Slack App

  1. 前往 https://api.slack.com/apps 創建新的 Slack App
  2. 選擇 “From an app manifest” 或 “From scratch”
  3. 設定名稱和要關聯的工作區

第二步：設定權限 Scopes

在 “OAuth & Permissions” 頁面，添加以下 Bot Token Scopes：

app_mentions:read, channels:history, channels:read, chat:write,
groups:history, groups:read, im:history, im:read, im:write,
mpim:history, mpim:read, mpim:write, reactions:read, reactions:write,
files:read, files:write, emoji:read, pins:read, pins:write, users:read

第三步：設定 Event Subscriptions

在 “Event Subscriptions” 頁面，訂閱以下 Bot Events：

app_mention, channel_rename, member_joined_channel, member_left_channel,
message.channels, message.groups, message.im, message.mpim,
pin_added, pin_removed, reaction_added, reaction_removed

注意事項

安全性考量

  1. Token 安全：App Token 和 Bot Token 都是敏感資訊，切勿將它們提交到公開的程式碼倉庫或與他人分享
  2. 環境變數：建議使用環境變數來存儲 token，而不是直接在配置文件中寫入
  3. 權限最小化：只授予必要的權限scope，不要添加不必要的權限
  4. DM Policy 選擇：、生產環境中建議使用 pairing 或 allowlist 模式，避免完全開放私訊權限

常見問題排查

問題：訊息沒有收到回覆

  • 檢查頻道是否在白名單中
  • 確認是否使用了 @mention 來觸發代理
  • 檢查 requireMention 設定是否正確
  • 查看 OpenClaw logs 確認是否有錯誤訊息

問題：無法建立 Socket 連接

  • 確認 App Token 正確且具有 connections:write 權限
  • 檢查網路連接是否正常
  • 查看 OpenClaw logs 中是否有錯誤訊息
  • 確認 Slack App 已經正確安裝到工作區

問題：權限不足

  • 確認已添加所有必要的 Scopes
  • 重新安裝 App 以確保權限生效
  • 檢查 Bot User 是否已啟用
  • 確認 Event Subscriptions 已啟用並驗證

生產環境建議

  1. 監控：定期檢查連接狀態和訊息處理延遲
  2. 日誌：保留足夠的日誌以便排查問題，建議設置日誌輪轉
  3. 備份：定期備份配置和 token 資訊
  4. 更新：關注 Slack API 的變化，及時更新權限和配置
  5. 測試：在正式環境部署前，先在測試環境驗證功能正常

優缺點

優點

  1. 雙模式連接：支援 Socket Mode 和 HTTP Events API，適合各種部署環境
  2. 細緻的權限控制：提供 DM 和頻道的完整權限管理
  3. 每頻道設定：可以為不同頻道配置不同的行為和技能
  4. Thread 支援：支援多輪對話，方便複雜問題的討論
  5. 串流回覆：支援即時串流，改善長回覆的使用者體驗
  6. 與 Slack 深度整合：支援所有 Slack 功能包括 Reaction、Pin 等

缺點

  1. 設定複雜：需要配置 Slack App、Scopes、Events 等，對新手不夠友善
  2. 依賴 Slack API：需要穩定的網路連接來與 Slack 通訊
  3. Token 管理：需要妥善管理 App Token 和 Bot Token，安全性要求較高
  4. Socket Mode 限制：需要 App Token 具有 connections:write 權限

與其他通訊工具比較

相較於 Telegram 通訊，Slack Integration 的優勢在於：

  • 支援更複雜的頻道架構和權限管理
  • 與企業工作流程整合更緊密
  • 支援更多的檔案和多媒體格式

劣勢在於：

  • 設定複雜度較高
  • 需要企業帳號才能使用完整功能
  • 行動端體驗相對一般

問題與解決

常見問題一：Socket Mode 連接失敗

問題描述：嘗試連接 Slack 時出現連接失敗的錯誤訊息

解決方案：

  1. 確認 App Token 正確且具有 connections:write 權限
  2. 檢查網路是否能夠訪問 Slack 的 WebSocket 端點
  3. 確認 Slack App 已經正確安裝到工作區
  4. 查看詳細錯誤訊息並根據提示修復

常見問題二：訊息無法觸發回覆

問題描述：發送訊息到頻道但沒有收到回覆

解決方案：

  1. 確認頻道 ID 在白名單中（如果使用 allowlist 模式）
  2. 確認使用了正確的 @mention 格式
  3. 檢查 requireMention 設定是否為 true
  4. 確認 Bot User 已經加入該頻道

常見問題三：權限被拒絕

問題描述：操作時出現權限不足的錯誤

解決方案：

  1. 檢查所有必要的 Scopes 是否已經添加
  2. 重新安裝 Slack App 以刷新權限
  3. 確認 Bot User 已經啟用
  4. 檢查 Event Subscriptions 是否已經啟用並驗證

常見問題四：串流回覆中斷

問題描述：使用串流回覆時，訊息發送不完整

解決方案：

  1. 檢查網路連接穩定性
  2. 嘗試降低串流速度或關閉串流
  3. 檢查 Slack 的速率限制
  4. 確認訊息內容沒有觸發 Slack 的內容審查
